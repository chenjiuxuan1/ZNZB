# 本次 Metabase 巡检改造完整说明

本文档记录从本次对话开始，到当前版本为止，对 Metabase 数据缺失巡检所做的主要调整。

目标只有一个：**只有 API 查询成功、规则时间已到、数据又确实不存在时，才报告数据缺失。**

## 1. 最开始的问题

原来的逻辑主要检查昨天或当天是否有数据，适合每天更新的看板，但会误报下面这些情况：

- 看板每 7 天才更新一次，却每天被要求必须有新数据。
- API 返回连续的每日明细，但这些明细其实是每周一次批量补齐。
- API 查询失败或链接 404，被页面混在“数据缺失”里展示。
- 不同国家时区不同，却可能按同一个时间理解 D0、D-1。
- 09:00 开始巡检，后处理的卡片可能按 09:02 或更晚的时间判断。
- 指标列改成英文后，规则仍找中文列名，导致“有值但被判空”。

## 2. 最终判断流程

一张卡片现在按下面的顺序处理：

1. 调用 Metabase API。
2. API 失败时记录“查询异常”，不证明数据缺失。
3. API 成功后识别真实日期列和指标列。
4. 按国家当地时区和本次巡检开始时间过滤未来数据。
5. 判断整表无行、指标全空、目标日期缺失或半小时时间点缺失。
6. 读取跨巡检观察记录，判断看板真正多久刷新一次。
7. 已确认刷新周期时，超过预计更新时间仍未推进才报警。
8. 证据不足时标记为 `learning`，不伪造刷新周期。

简单理解：

```text
API 成功
  → 时间已经到了
  → 按真实刷新周期本来应该更新
  → 原始 API 行里仍然没有
  → 才是数据缺失
```

## 3. 数据粒度和刷新周期已经拆开

以前容易把“结果里每天一行”理解成“看板每天刷新”。现在拆成两个概念：

- `dataCadence`：API 单次返回结果的行粒度，例如每日一行、每周一行。
- `refreshCadence`：多次巡检中，最新业务日期真正推进的频率。

例如一个看板每周一刷新，但一次补齐过去 7 天的每日数据：

- `dataCadence` 是每天。
- `refreshCadence` 是每周。
- 缺失判断应该按每周刷新，而不是每天报警。

## 4. 刷新周期如何学习

系统会观察每次巡检中 `latestBusinessDate` 是否变化，只记录真正的变化事件。

目前支持识别：

- 自然日每日更新。
- 工作日每日更新。
- 每 N 天更新。
- 每 N 周更新。
- 每月更新。
- 每 2 个月或每 3 个月更新。

确认周期至少需要：

- 4 个变化事件。
- 3 个有效间隔。
- 置信度不低于 75%。

证据不足或规律混乱时保持 `learning`。现有明确配置的 D0/D-1 卡片在学习期继续沿用原规则；自动发现但尚未学成的卡片只观察，不强制报警。

观察缓存采用原子写入，默认保留 400 天，每张卡片最多保留 500 个事件，并已加入 `.gitignore`，不会作为业务代码提交。

## 5. 预计到达时间和宽限期

系统不仅学习“哪一天更新”，还会记录数据通常在当地几点出现。

预计到达时间使用历史到达时间的 P90，并额外增加 2 小时宽限。这样偶尔晚一点的数据不会立刻报警。

周末会根据历史自动识别；节假日默认不猜测，后续可以通过国家级 `holidayDates` 配置补充。

## 6. 所有规则使用国家当地时间

规则中的 `timezone: "dashboard"` 会使用国家配置：

| 国家 | 时区 |
|---|---|
| 中国 | `Asia/Shanghai` |
| 印尼 | `Asia/Jakarta` |
| 菲律宾 | `Asia/Manila` |
| 泰国 | `Asia/Bangkok` |
| 巴基斯坦 | `Asia/Karachi` |
| 墨西哥 | `America/Mexico_City` |

例如北京时间 7 月 21 日 09:00 执行时，墨西哥还是 7 月 20 日晚上。因此墨西哥的 D0 是当地 7 月 20 日，不是北京时间的 7 月 21 日。

使用的是 IANA 时区名称，不是写死的 UTC 偏移，因此会按时区数据库处理当地时间规则。

## 7. 一次巡检固定使用同一个时间

巡检开始时记录一次 `checkedAt`，之后所有国家、卡片和规则都使用这个时间。

例如 09:00 开始巡检，即使最后一张卡片到 09:04 才处理，仍按 09:00 判断，避免不同卡片因为处理顺序不同而多检查一个时间点。

## 8. 所有规则都会过滤未来数据

时间过滤放在规则统一入口，不需要每条规则重复配置：

- 未来日期不参与判断。
- 当天有时间列时，只保留执行时刻及以前的数据。
- 配置截止时间晚于执行时间时，以执行时间为准。
- 配置截止时间更早时，以更早的配置时间为准。
- 半小时规则配置 60 分钟延迟时，09:00 执行只检查到 08:00。

半小时缺点还会和前一天已有时间点对齐。只有前一天该点存在、今天该点不存在，并且已经超过允许延迟，才报告缺失。

## 9. 日期列和指标列识别修复

日期列优先使用规则中真实存在且包含有效日期值的字段，否则依次尝试看板维度和数据值推断。

特别修复了把 `开始时间=08:30` 错当成业务日期的问题。

指标列优先使用配置中真实存在的字段；配置字段不存在时回退到卡片实际 metrics，并记录 `schemaMismatch`，但不会因此把有值的数据判空。

泰国“用户还款后 7 日复借率”已经可以识别：

- 日期列 `finish_date`。
- 英文指标列，例如 `repay_cnt`、`freeze_rate_7`。

## 10. 0 不再被当成空值

- 数值 `0` 是有效业务值。
- `null`、空字符串和字段不存在才算空。
- 有行但所有有效指标都为空时，才是 `emptyMetrics`。
- 完全没有行时是 `noData`。

这些状态不会再混成一个模糊的“数据缺失”。

## 11. 中国停业卡片已加入忽略名单

下面 7 张中国卡片业务已经停止，值为 0 属于正常情况，现在不会参与正式告警：

- 新客-通过次数
- 老客-通过次数
- 新客-irr放款金额
- 老客-irr放款金额
- 新客-apr放款金额
- 老客-apr放款金额
- 老客-irr还款金额

忽略规则允许标题末尾存在空格，避免因为 Metabase 标题格式差异漏匹配。

## 12. 墨西哥 404 看板已修复

原墨西哥“放款统计”使用的 public UUID 已失效，导致 12 张卡片全部返回 404。

根据最新报表清单，来源已改为内部看板：

```text
dashboardId = 280
title = 资产管理-放款统计
uuid = internal-280
```

查询会使用 `dashboardId + cardId + dashcardId`，不再访问失效的 public UUID。服务器需要配置 Metabase 内部登录态才能访问内部看板。

## 13. 查询异常和数据缺失已经分开

502、503、504 通常表示 Metabase、网关或底层查询临时不可用，不代表业务没有数据。

公开和内部 Metabase API 现在都会：

1. 第一次正常请求。
2. 502/503/504 时自动重试 2 次。
3. 共 3 次仍失败，才记录查询异常。

404、权限失败等永久性错误不会重复请求。

页面现在分别显示：

- `数据缺失`：API 成功返回，但目标数据确实不存在。
- `查询异常`：API 没有成功返回，例如 502、权限失败或链接失效。

墨西哥“每期逾期率”曾出现一次 502。随后使用相同参数连续重试均成功：默认查询返回 35 行，130 天历史查询返回 124 行，因此该次 502 属于临时查询故障，不是数据缺失。

## 14. 历史日期中间缺口

系统不只检查最新日期，还会检查最近 7 个预计数据点。

因此“最新日期已经存在，但中间漏了一天”也可以识别为 `historicalDateGap`。

## 15. 输出结果增加了刷新证据

`checkedCards` 中增加 `freshness`：

```json
{
  "freshness": {
    "status": "learning|healthy|overdue|schema_error",
    "dateColumn": "finish_date",
    "metricColumns": ["repay_cnt", "freeze_rate_7"],
    "latestBusinessDate": "2026-07-21",
    "dataCadence": { "label": "每天" },
    "refreshCadence": { "label": "每周", "confidence": 0.9 },
    "nextExpectedAt": "2026-07-28T10:00:00+07:00",
    "evidenceCount": 5
  }
}
```

旧的 `updateCadences` 字段继续保留，兼容已有页面和结果消费者；内容表示已经确认的真实刷新周期。

## 16. 验证方式和结果

本次改造增加并执行了以下验证：

- 每日、工作日、每 2 天、每周、每月和季度刷新样本。
- 每周一次刷新但批量补齐 7 天每日明细。
- 日期字段中英文、时间列误识别、英文指标回退。
- 无行、指标全空、指标为 0、目标日期缺失和中间日期断档。
- 不同国家时区、D0/D-1、执行时间截止和未来数据过滤。
- 观察缓存损坏、过期和原子写入。
- 502/503/504 重试以及 404 不重试。
- 平台 API、通知和既有规则回归测试。

当前全量自动化测试结果为 **157/157 通过**。

真实 Metabase API 定向比对中：

- 31 张半小时缺失卡片。
- 共 79 个缺失时间点。
- 79 个点均确认“前一天存在、当天不存在”。
- API 比对不一致为 0。
- 153 次目标日期检查全部存在。

本地没有内部 Metabase 登录态时，中国内部看板和墨西哥内部看板 280 会显示访问配置问题，这类结果不会当成数据缺失。

## 19. 排查告警时先看什么

1. 先看异常类型是“数据缺失”还是“查询异常”。
2. 查询异常先看重试结果，不要直接判断业务没数据。
3. 数据缺失要核对国家当地日期和本次巡检的 `checkedAt`。
4. 半小时缺点要确认该点在前一天存在、当天不存在，并早于截止时间。
5. 数值为 0 时确认它是正常业务值还是业务已停止。
6. 周更或月更看板要看 `refreshCadence` 和 `evidenceCount`，不要只看单次 API 的日期分布。
7. 内部看板失败时先检查服务器是否配置 `METABASE_SESSION`、`METABASE_COOKIE` 或 `METABASE_API_KEY`。
