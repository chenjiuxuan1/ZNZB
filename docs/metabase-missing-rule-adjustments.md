# Metabase 数据缺失规则调整说明

这份文档用简单方式说明巡检现在如何判断“数据缺失”，以及本次调整解决了什么问题。

## 1. 每个国家使用自己的当地时间

巡检虽然由同一台服务器执行，但不会把北京时间直接套给所有国家。规则中的 `timezone: "dashboard"` 会使用国家配置的时区：

| 国家 | 时区 |
|---|---|
| 中国 | `Asia/Shanghai` |
| 印尼 | `Asia/Jakarta` |
| 菲律宾 | `Asia/Manila` |
| 泰国 | `Asia/Bangkok` |
| 巴基斯坦 | `Asia/Karachi` |
| 墨西哥 | `America/Mexico_City` |

例如北京时间 7 月 21 日 09:00 执行时，墨西哥还是 7 月 20 日晚上。因此墨西哥的 D0 是当地 7 月 20 日，不是北京时间的 7 月 21 日。

## 2. 一次巡检只使用一个固定时间

巡检开始时会记录 `checkedAt`，所有卡片和所有规则都使用这个时间。

例如 09:00 开始巡检，即使最后一张卡片到 09:04 才处理，也仍按 09:00 判断，不会多检查 09:00 之后才应该出现的数据。

## 3. 不检查执行时间之后的数据

所有规则在判断前都会过滤未来数据：

- 未来日期不参与判断。
- 当天带时间列的数据，只保留执行时刻及以前的时间点。
- 如果规则配置了更早的截止时间，使用更早的时间。
- 半小时规则配置了 60 分钟延迟时，09:00 执行只检查到 08:00。

## 4. 每日、每周和每月更新分开判断

单次 API 返回很多连续日期，不代表看板每天刷新。系统会记录多次巡检中“最新业务日期何时推进”，至少获得 4 个变化事件后，才确认真实刷新周期。

- 已确认每天刷新：按天判断是否逾期。
- 已确认每周刷新：到预计周更时间后仍未更新才报警。
- 已确认每月刷新：到预计月更时间后仍未更新才报警。
- 证据不足：显示 `learning`，不会把猜测当成刷新周期。

## 5. 502、503、504 不等于数据缺失

这些状态通常表示 Metabase、网关或底层查询临时不可用。现在公开和内部 Metabase API 都会自动重试 2 次，三次都失败后才记录查询异常。

页面会把结果明确分开：

- `数据缺失`：API 查询成功，但目标日期、时间点或有效指标确实不存在。
- `查询异常`：API 没有成功返回，例如 502、权限失败或链接失效。

查询异常不会作为 API 已证明的数据缺失。

## 6. 数值 0 和空值的区别

- `0` 是有效业务值，不算空。
- `null`、空字符串和字段不存在才算空。
- 已明确停业的中国卡片在忽略名单中，不参与正式告警。

## 7. 排查告警时先看什么

1. 先看异常类型是“数据缺失”还是“查询异常”。
2. 查询异常先重试 API，不要直接判断业务没数据。
3. 数据缺失要核对国家当地日期和执行截止时间。
4. 半小时缺点要确认该点在前一天存在、当天不存在，并且早于截止时间。
5. 周更或月更看板要等真实刷新周期学习完成后再判断逾期。
